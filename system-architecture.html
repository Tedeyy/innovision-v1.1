Innovision System Architecture Overview

This document outlines the high-level architecture of the Innovision platform, covering clients, backend services, data stores, and integrations, including IPROGSMS for notifications.

Clients (Web)
 ├─ BAT Dashboard (PHP)
 ├─ Admin Dashboard (PHP)
 └─ Superadmin Dashboard (PHP)

Backend (PHP application layer)
 ├─ Session/Auth (Supabase Auth + PHP sessions)
 ├─ REST Helper (sb_rest): PostgREST to Supabase tables
 ├─ Business Modules
 │   ├─ User Reviews & Approvals
 │   ├─ Listing Management & Reviews
 │   ├─ Reports & Penalties
 │   ├─ Pricing (Market price proposals)
 │   └─ Analytics (Charts)
 └─ Integrations
     ├─ IPROGSMS (SMS notifications)
     └─ Leaflet (Maps), Chart.js (Charts)

Data Layer (Supabase / Postgres)
 ├─ Users: buyer, seller, bat, admin, superadmin
 ├─ Review Queues: reviewbuyer, reviewseller, reviewbat, reviewadmin
 ├─ Listings: activelivestocklisting, reviewlivestocklisting
 ├─ Reports: reviewreportuser, reportuser
 ├─ Penalties: penalty
 ├─ Pricing: livestock_type, livestockbreed, marketprice(pending/approved)
 └─ Logs: livestocklisting_logs, action logs

Flows
 1) Registration/Verification
    client → review tables → admin/superadmin actions → main tables
 2) Listing Review & Publication
    seller → reviewlivestocklisting → admin/BAT approves → activelivestocklisting
 3) Reporting & Penalties
    client → reviewreportuser → admin verifies → reportuser/penalty entries
 4) Pricing Governance
    proposal → pending → approval → market price update
 5) Analytics
    queries from Supabase → Chart.js visualizations
 6) Notifications (IPROGSMS)
    app events → SMS compose → IPROGSMS API → recipient

Clients

  • Responsive PHP-based dashboards (BAT, Admin, Superadmin)
  • Leaflet for maps; Chart.js for visualizations
  • Mobile-first CSS for compact views

Backend

  • PHP controllers/pages call sb_rest to access Supabase (PostgREST)
  • Session-based user state; role-aware UI and logic
  • Context-aware modals for actions (approve/deny/penalty)

Data Model

  • Separation of review vs. production tables
  • Audit trails via logs (e.g., livestocklisting_logs)
  • Pricing references: livestock_type, livestockbreed

Notifications (IPROGSMS)

  • Use-cases: approvals, report updates, penalties, password resets
  • Outbound: server-side PHP calls to IPROGSMS REST API
  • Recommended: queue and retry, per-recipient rate limiting
  • Template variables: recipient name, action links, codes

Security

  • Supabase Row-Level Security (RLS) on data access
  • Server-side validation for admin actions
  • JWT/session handling; CSRF-safe posts

Scalability

  • Stateless PHP pages backed by Supabase REST
  • Cache read-mostly lists (types, breeds) client-side
  • Background workers for heavy tasks and SMS dispatch

IPROGSMS Integration (Example)

POST https://api.iprogsms.example.com/v1/messages
Authorization: Bearer <API_KEY>
Content-Type: application/json

{
  "to": "+639XXXXXXXXX",
  "text": "Your report was approved. Ref: {{ref}}",
  "senderId": "INNOVISION"
}
  • Store API key securely (env/secret), never in client-side code.
  • Wrap in a PHP helper (e.g., send_sms_via_iprogsms($to,$text)).
  • Log request/response for auditing and retries.

Deployment Considerations

  • Environment configs for Supabase and IPROGSMS
  • HTTPS everywhere; HSTS; secure cookies
  • Backups for critical tables; rotation for logs
